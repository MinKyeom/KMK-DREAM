DB Lock 개념 및 종류 정리

데이터베이스(DB)에서 **Lock(락)**은 여러 사용자가 동시에 같은 데이터에 접근할 때, 데이터의 일관성과 무결성을 유지하기 위해 자원 사용을 제한하는 메커니즘입니다.

쉽게 비유하자면, 화장실 칸에 들어가 문을 잠그는 것과 같습니다. 내가 사용 중일 때 다른 사람이 들어오지 못하게 막아 '데이터 뒤섞임'을 방지하는 것이죠.

1. Lock의 주요 종류
데이터베이스는 상황에 따라 크게 두 가지 종류의 락을 사용합니다.

공유 락 (Shared Lock, S-Lock)
용도: 데이터를 읽을 때(SELECT) 사용합니다.

특징: 여러 명이 동시에 공유 락을 거는 것은 허용됩니다. 즉, "나도 읽고 너도 읽는 것"은 문제가 없기 때문입니다.

제한: 누군가 공유 락을 걸고 있다면, 다른 사람이 수정(Exclusive Lock)할 수는 없습니다.

배타 락 (Exclusive Lock, X-Lock)
용도: 데이터를 쓸 때(INSERT, UPDATE, DELETE) 사용합니다.

특징: "나만 쓸 거야"라는 모드입니다.

제한: 배타 락이 걸린 자원에는 다른 누구도 접근(읽기/쓰기 모두)할 수 없으며, 이미 다른 락이 걸려 있는 자원에 배타 락을 걸 수도 없습니다.

2. Lock의 설정 범위 (Granularity)
락을 어디까지 걸 것인가에 따라 성능과 관리 비용이 달라집니다.

Database Lock: 데이터베이스 전체를 잠급니다. (주로 업데이트/점검 시 사용)

Table Lock: 테이블 전체를 잠급니다.

Page/Block Lock: 데이터가 저장된 물리적 블록 단위로 잠급니다.

Row Lock: 특정 행(Row) 하나만 잠급니다. 현대 DB에서 가장 많이 쓰이며 정밀하지만 관리 비용이 높습니다.

3. Lock으로 인해 발생하는 문제들
락은 데이터를 보호하지만, 잘못 관리하면 시스템이 멈추는 부작용이 생깁니다.

데드락 (Deadlock, 교착 상태)
두 개 이상의 트랜잭션이 서로가 가진 락을 해제하기만을 기다리며 무한 대기에 빠지는 상황입니다.

예: A는 1번을 잡고 2번을 기다림, B는 2번을 잡고 1번을 기다림.

블로킹 (Blocking)
먼저 락을 획득한 트랜잭션이 작업을 마칠 때까지 뒤의 트랜잭션들이 멈춰있는 상태입니다. 성능 저하의 주범이 되므로 트랜잭션을 최대한 짧게 유지하는 것이 중요합니다.

4. 낙관적 락 vs 비관적 락
개발자가 락을 다루는 전략적 관점에서의 구분입니다.

구분	비관적 락 (Pessimistic Lock)	낙관적 락 (Optimistic Lock)
가정	"데이터 충돌이 반드시 일어날 거야."	"데이터 충돌이 거의 안 일어날 거야."
방식	데이터를 읽을 때부터 물리적 락을 겁니다.	락을 걸지 않고, 수정 시점에 데이터 버전(Version)을 체크합니다.
장점	데이터 무결성이 확실히 보장됨.	별도의 락 비용이 없어 성능상 유리함.
단점	동시성이 떨어지고 데드락 위험이 있음.	충돌 발생 시 롤백 처리를 직접 구현해야 함.
정리하자면, DB Lock은 데이터의 정확성을 지키는 수호자이지만, 너무 남용하면 성능이 느려지고 아예 멈춰버릴 수도 있는 양날의 검입니다.
